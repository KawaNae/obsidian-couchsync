# モバイル編集中の巻き戻り問題 — 調査レポート

**報告日:** 2026-04-05
**報告者:** Claude Opus 4.6 (brainstorming session)
**対象:** obsidian-couchsync v0.5.3 (develop branch, commit bdec1d9)

---

## 1. 現象

モバイル端末（iPad / Android）でsyncをオンにしながらノートを編集すると、入力中のテキストが数秒〜数十秒前の状態に巻き戻る。

### 再現条件
- モバイル端末でObsidian + CouchSyncを使用し、syncオン状態で編集
- PC側のObsidianも起動中（ただしPC側では編集していない）
- モバイルネットワーク（LTE/5G）接続
- iPad、Android両方で複数回再現を確認

### ユーザーが観測した事象
- 編集中にリアルタイムで巻き戻りを目撃（バックグラウンド復帰時ではない）
- Obsidian本体から「外部で編集が入ったためマージします」という日本語通知が表示
- 通知後、入力中のテキストが以前の状態に戻る（例：「今日は良い感じだった」→「きょうｈ」）

---

## 2. 関連コードの現状

### 2.1 `dbToFile` — リモート→ローカルの書き込み (`vault-sync.ts:57-97`)

リモートから受信したドキュメントをローカルvaultに書き込む処理。

**問題点:**
- **内容比較を行わない。** 受信した内容が現在のファイル内容と同一でも、古くても、無条件で `vault.modify()` を実行する
- **アクティブ編集中のファイルかどうかを判定しない。** ユーザーがエディタで開いて入力中のファイルにも同様に書き込む
- **`vault.modify()` を使用している。** これはObsidianが「外部からのファイル変更」として検知するAPIであり、エディタで開いているファイルに対して使うとObsidian本体のマージ処理が発動する

### 2.2 `fileToDb` — ローカル→リモートの送信 (`vault-sync.ts:21-55`)

ローカルファイルの変更をCouchDBに送信する処理。

**備考:**
- チャンクIDベースの内容比較があり（36-42行目）、内容が同一ならスキップする
- ただしこの比較は送信側のみに存在し、受信側（`dbToFile`）には存在しない

### 2.3 change-trackerのpause/resume (`change-tracker.ts:65-75`, `main.ts:75-87`)

リモート変更の適用中にローカル変更のトラッキングを一時停止する仕組み。

**問題点:**
- pause期間が500ms固定。Obsidianのファイル処理（キャッシュ更新、インデクシング等）がこの時間内に完了しない場合、pause解除後にObsidianが触ったファイルのmodifyイベントがchange-trackerに拾われる
- pause中に発生したローカル編集はpendingQueueに蓄積されるが、resume時に一括で`scheduleSync`される。このタイミングで古い内容と新しい内容が競合する可能性がある

### 2.4 衝突解決 (`conflict-resolver.ts`)

CouchDBの `_conflicts` を検出し、mtimeベースで勝者を決定する。

**問題点:**
- **mtime laundering（タイムスタンプ洗浄）が起きる。** PCが受信したドキュメントを `dbToFile` でvaultに書き込むと、その時点のディスク書き込み時刻が新しいmtimeになる。この「PC経由で更新されたmtime」を持つ古い内容が、スマホ側の新しい内容より新しいmtimeを持つことがある → 古い内容が衝突解決で勝者になる
- mtimeは内容の新しさではなく「最後にディスクに書き込まれた時刻」であり、中継デバイスを経由すると信頼性が失われる

### 2.5 reconnectWithPullFirst (`main.ts:229-270`)

ネットワーク復帰時にリモートから一括pullして適用する処理。

**問題点:**
- pullで取得した全ドキュメントを `dbToFile` で適用するが、ローカルの現在の内容との比較を行わない
- モバイルネットワークは断続的に切断・再接続を繰り返すため、この処理が頻繁に発火する
- 再接続のたびにリモートの（古い可能性がある）内容でローカルファイルを上書きするリスクがある

### 2.6 debounce設定 (`change-tracker.ts:87`)

ローカル変更のCouchDBへの送信は2000ms（2秒）のdebounceで制御されている。

**備考:**
- 2秒間隔でdebounceが発火するため、入力途中の不完全な内容がCouchDBに送信される
- これ自体は正常な動作だが、上記の問題と組み合わさると「途中状態がPCにechoされて衝突を起こす」というシナリオの起点になる

## 3. 推定される巻き戻りシナリオ

### シナリオA: PC経由のecho衝突（最も可能性が高い）

```
時刻  スマホ                  CouchDB              PC
─────────────────────────────────────────────────────────
T+0s  入力開始「き」
T+2s  debounce発火            rev 2:「きょう」←push
      「きょう」を送信                              pull→dbToFile
                                                   「きょう」をvaultに書込
T+2.5s                                             pause解除
T+3s  入力継続                                     Obsidianが何らかの
      「今日は良い感じ」                             ファイル処理→modify
                                                   changeTracker発火→
                                                   fileToDb（※）
T+4s  debounce発火            rev 3:「今日は良い…」  rev 3':「きょう」
      「今日は良い感じだった」                        ↑ 衝突発生！
      を送信
T+5s  pull←衝突解決           衝突解決:
      dbToFile:                mtime比較→
      vault.modify(「きょう」)  PC側のmtimeが新しい
      Obsidian:「外部編集       → 古い内容が勝者
      マージします」
      → 巻き戻り！
```

※ `fileToDb`のチャンク比較がPCのObsidianによるファイル変更（メタデータ等）を検出した場合

### シナリオB: モバイルネットワーク再接続

```
時刻  スマホ                  CouchDB
─────────────────────────────────────────
T+0s  入力中「今日は良い感じ」
T+1s  ネットワーク瞬断
      → handleOffline()
T+2s  ネットワーク復帰
      → handleOnline()
      → reconnectWithPullFirst()
      → pullFromRemote()
T+3s  pull完了                 古いrev返却
      dbToFile: vault.modify()
      → Obsidian「外部編集マージ」
      → 巻き戻り！
```

### シナリオC: A+Bの複合

モバイルネットワークの不安定さがecho衝突の頻度を増幅する。再接続のたびにpull-firstが発火し、PC経由のecho revisionを受け取る。

## 4. 問題の要約

| # | 問題 | 該当箇所 | 影響度 |
|---|------|----------|--------|
| 1 | `dbToFile`が内容比較なしで無条件書き込み | `vault-sync.ts:57-97` | **致命的** |
| 2 | `vault.modify()`がObsidianの外部編集検知を発火 | `vault-sync.ts:90` | **致命的** |
| 3 | 衝突解決のmtime比較がPC中継で信頼性を失う | `conflict-resolver.ts` | **高** |
| 4 | change-trackerのpause時間500msが不十分な場合がある | `main.ts:80-83` | **中** |
| 5 | `reconnectWithPullFirst`が内容比較なしで全適用 | `main.ts:248-251` | **高** |
| 6 | アクティブ編集中のファイルに対する保護がない | 全体 | **致命的** |

---

## 5. 影響範囲

- **モバイル端末での編集全般** — PCが起動している限り、どのファイルでも発生しうる
- **モバイルネットワーク環境で特に顕著** — 再接続頻度が高いほど発生確率が上がる
- **日本語IME入力で特に体感しやすい** — debounce発火時にIMEの変換途中状態が送信され、echoされた際に明らかな巻き戻りとして認識される
